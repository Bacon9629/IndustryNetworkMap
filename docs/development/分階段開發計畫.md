# 台股產業鏈知識圖譜 Web App — 分階段開發計畫

## 1. 開發計畫總覽

本計畫將專案切分為多個小階段，避免一開始就進入過大的 phase。

整體順序為：

```text
Phase 0：專案邊界與資料結構定義
Phase 1：Graph Database 基礎建設
Phase 2：Seed Data 與匯入流程
Phase 3：Backend Graph API
Phase 4：Frontend Focus Graph
Phase 5：公司 / 產品 / 產業查詢體驗
Phase 6：Demand Shock 初版分析
Phase 7：資料品質與審核流程
Phase 8：擴充至台股前 100
Phase 9：RAG / Agent 資料抽取
Phase 10：產品化與進階分析
```

---

* 建立 `docs/development/relationship-types.md`

產出：

```text
docs/
├── design.md
├── data-model.md
├── relationship-types.md
└── roadmap.md
```

完成標準：

* 專案目標清楚。
* 非目標清楚。
* 初期 scope 清楚。
* 後續開發可以依文件推進。

---

### Phase 0.2：定義核心節點類型

初期節點類型：

完成標準：

* 每種 node 的必要欄位定義完成。
* 每種 node 的 optional 欄位定義完成。
* 每種 node 的命名規則定義完成。
* 每種 node 的 id 規則定義完成。

### Phase 0.3：定義核心關係類型

初期 relationship types：

```text

PRODUCES
USES
```

完成標準：

* 每種 relationship 的 from / to 類型定義完成。
* 每種 relationship 的語意定義完成。
* 每種 relationship 的方向定義完成。

```text
rejected
```

## Phase 1：Graph Database 基礎建設

目標：

> 建立 Neo4j，讓系統有正式 graph 儲存層。

---

### Phase 1.1：建立 Neo4j 開發環境

工作項目：

* 使用 Docker Compose 建立 Neo4j。
* 設定帳號密碼。
* 設定 data volume。
* 建立本機開發啟動方式。

完成標準：

* 可以本機啟動 Neo4j。
* 可以進入 Neo4j Browser。
* 可以執行基本 Cypher query。

---

### Phase 1.2：建立 Neo4j Constraints

工作項目：

* Company.id unique。
* Product.id unique。
* Industry.id unique。
* Application.id unique。
* Source.id unique。
* Evidence.id unique。

完成標準：

* 重複 id 無法寫入。
* 所有核心 node 都有唯一 id。

---

### Phase 1.3：建立測試 graph

工作項目：

* 手動建立 3 家公司。
* 手動建立 3 個產品。
* 手動建立 1 個應用。
* 手動建立幾條關係。

範例：

```text
台積電
NVIDIA
AI Server
AI GPU
```

完成標準：

* Neo4j 中可以看到最小 graph。
* 可以查詢 1-hop 關係。

---

## Phase 2：Seed Data 與匯入流程

目標：

> 建立可重複執行的資料匯入流程，不依賴手動在 Neo4j Browser 建資料。

---

### Phase 2.1：定義 seed file 格式

建立：

```text
seed_companies.csv
seed_products.csv
seed_industries.csv
seed_applications.csv
seed_sources.csv
seed_relationships.csv
```

完成標準：

* 每個 CSV 欄位定義清楚。
* 欄位名稱與 data model 對齊。
* 所有 id 命名規則一致。

---

### Phase 2.2：建立第一批 seed data

建議初期資料量：

```text
公司：20 到 30 家
產品：20 個
產業：10 個
應用：10 個
關係：80 到 120 條
```

完成標準：

* 至少有一個完整的 AI Server 供應鏈範例。
* 至少有一個公司可以查到上游與下游。
* 至少有一個產品可以查到相關公司。
* 至少有一個應用可以查到相關產品。

---

### Phase 2.3：建立 import script

工作項目：

* 讀取 seed CSV。
* 驗證欄位。
* 寫入 Neo4j。
* 支援清空後重建。
* 支援重複執行。

完成標準：

* 執行 script 後可完整建立 graph。
* 重複執行不會產生重複節點。
* 匯入失敗時能顯示錯誤原因。

---

### Phase 2.4：建立資料驗證 script

驗證項目：

* node id 不重複。
* relationship from 存在。
* relationship to 存在。
* relationship type 合法。
* confidence 範圍合法。
* status 合法。
* source id 存在。
* 必要欄位不為空。

完成標準：

* 錯誤資料無法匯入。
* 錯誤訊息可以定位到 row。

### Phase 3.1：建立 FastAPI 專案

* Product
* Industry

完成標準：

完成標準：

* 可取得 Company detail。
* 可取得 Product detail。
* 可取得 Industry detail。
* 可取得 Application detail。

---

### Phase 3.4：建立 Neighborhood Graph API

```text

```


完成標準：


```text

```

完成標準：

---

### Phase 6.2：建立 Graph Traversal Query

工作項目：

* 找相關 Product。

完成標準：

* 每家公司有 score。
* 每家公司有至少一條傳導路徑。

* evidence。

完成標準：

## Phase 10：產品化與進階分析

目標：

> 從 MVP 變成真正可用的投資研究工具。

---

### Phase 10.1：進階 Graph Analysis

可能功能：

* 二階供應鏈分析。
* 關鍵節點分析。
* 供應鏈集中度分析。
* 產業瓶頸分析。
* 公司暴露度分析。

---

### Phase 10.2：進階 Demand Shock

可能功能：

* 需求增加。
* 需求下降。
* 原物料漲價。
* 供應中斷。
* 地緣政治風險。
* 匯率影響。
* 利率影響。

---

### Phase 10.3：自然語言查詢

使用者可輸入：

```text
如果 AI Server 需求增加，台股有哪些公司可能受益？
```

系統流程：

```text
LLM 解析問題
    ↓
轉成 graph query
    ↓
執行 traversal
    ↓
RAG 找 evidence
    ↓
LLM 生成解釋
```

---

### Phase 10.4：3D Graph Exploration

若 2D 功能成熟，可加入 3D 模式。

原則：

* 3D 是輔助探索，不取代 2D 分析。
* 3D 不應成為主要研究介面。
* 3D 適合展示整體產業網絡，但不適合細節查證。

---

## 建議立即執行的下一步

最適合現在開始的工作是 Phase 0。

具體順序：

```text
1. 建立專案 repo。
2. 建立 docs/。
3. 放入 design.md。
4. 放入 roadmap.md。
5. 另外建立 data-model.md。
6. 另外建立 relationship-types.md。
7. 決定第一批 20 到 30 家 seed 公司。
8. 決定第一批產品、產業、應用節點。
9. 寫出 seed CSV 格式。
10. 再開始建立 Neo4j。
```

短期內最重要的不是寫 UI，而是先鎖定：

```text
node schema
relationship schema
relationship type taxonomy
seed data format
data validation rules
```

這四個穩定後，後面的 Backend、Frontend、RAG、Agent 才會接得起來。
