# 新增與更新公司資料指南

情境：手邊有幾家公司的財報（年報 / 法說會簡報 / 官網等），想把裡面揭露的供應鏈關係
（供應商、代工廠、客戶、使用的零組件或設備…）放進本系統的 Neo4j graph。

本文件只講「怎麼做」；規則與 schema 的權威定義在：

- [docs/development/data-model.md](docs/development/data-model.md)：節點欄位、id 規則、confidence/status 定義
- [docs/development/relationship-types.md](docs/development/relationship-types.md)：關係類型白名單與方向
- [docs/development/data-update-rules.md](docs/development/data-update-rules.md)：狀態轉換、同一關係跨期間怎麼處理
- [docs/usage.md](docs/usage.md)：安裝與各頁面操作

---

## 0. 先選路徑：手動 seed CSV，還是讓 Agent（RAG pipeline）抽取？

| | 手動寫入 `ingestion/seeds/*.csv` | Agent / RAG 半自動抽取 |
|---|---|---|
| 適合情況 | 你已經自己讀完財報、關係數量少（一次幾條到幾十條）、想立即生效 | 你手邊是原始 PDF/文字文件，想讓 Agent 先讀過並列出候選關係，你再審核 |
| 產出狀態 | 可直接 `status=verified`（因為是你本人確認後才寫入） | 一律 `status=candidate`，必須到 `/review` 頁人工 Accept 才會生效於正式分析 |
| 可信度紀錄 | 你自己在 CSV 填 `source_ids` / `confidence` | Agent 自動附 evidence 逐字引用 + confidence，來源可追溯 |
| 新公司（graph 裡沒有的） | 直接在 `seed_companies.csv` 新增一列 | Agent 會嘗試建立候選節點，若資訊不足（缺 ticker/交易所）仍需你手動補 |

兩種路徑最終都會寫進同一個 Neo4j，可以混用：例如 5 家公司裡，資訊明確的直接手動寫 seed，
內容複雜或篇幅長的財報交給 Agent 抽取再審核。

**鐵律（兩種路徑都適用）**：每條關係都要有可追溯的來源（`source_ids` 指向一個 `Source` 節點）；
沒有明確依據就不要寫（允許 `unknown`，不允許亂猜數字或方向）。

## 1. 共用前置作業：把財報登記成 Source

不論走哪條路徑，財報本身都要先在系統裡登記成 `Source` 節點，關係才有東西可以引用。

### 1.1 手動路徑：直接編輯 `ingestion/seeds/seed_sources.csv`

新增一列，例如：

```csv
source_tsmc_2025_annual_report,台積電 2025 年報,annual_report,https://www.tsmc.com/...,台積電,2026-03-15,2026-07-05,TWSE_2330,2025,zh-TW
```

欄位：`id, title, type, url, publisher, published_date, retrieved_at, company_id, period, language`。
`type` 只能是 `annual_report` / `financial_report` / `investor_presentation` / `company_website` /
`mops` / `exchange_data` / `news` / `research_report` / `manual` 其中之一。

### 1.2 Agent 路徑：放檔案 + 登記 manifest

```bash
cd ingestion
# 1. 把財報檔案（txt / html / pdf）放進 rag/documents/
copy "你的台積電年報.pdf" rag/documents/tsmc_2025_annual_report.pdf

# 2. 在 rag/documents/manifest.csv 新增一列
# file, source_id, company_id, period, type, url, title, language
```

```csv
tsmc_2025_annual_report.pdf,source_tsmc_2025_annual_report,TWSE_2330,2025,annual_report,https://www.tsmc.com/...,台積電 2025 年報,zh-TW
```

`company_id` 留空也可以（例如財報同時提到多家公司），但填了可以幫助 Agent 判斷上下文。
若是**全新公司**、graph 裡還沒有它的 id，`company_id` 先留空即可。

登記完執行：

```bash
python rag/register_sources.py     # manifest → MERGE Source 節點（不需 OPENAI_API_KEY）
python rag/parse_documents.py      # 解析 + chunking → rag/chunks/chunks.jsonl（不需 key）
```

## 2. 情境 A：全新公司（graph 裡完全沒有登記過）

例如財報裡提到一家從沒出現過的供應商。

### 路徑 1：手動（你已經確認好資料）

1. 在 `ingestion/seeds/seed_companies.csv` 新增一列（id 規則見 data-model.md：
   台股上市 `TWSE_xxxx`、上櫃 `TPEX_xxxx`、海外 `{國別/交易所}_{TICKER}`）：

   ```csv
   TWSE_3037,欣興電子,Unimicron Technology,3037,TWSE,Taiwan,true,Unimicron,https://www.unimicron.com,ABF載板與PCB大廠,2026-07-05
   ```

2. 在 `seed_relationships.csv` 新增這家公司相關的關係列，`source_ids` 指向你剛登記的 Source
   （見上一節 1.1），`status=verified`、`confidence` 依 data-model.md 的區間自行評估。
3. 驗證並匯入：

   ```bash
   cd ingestion
   python validators/validate.py
   python scripts/import_graph.py
   ```

### 路徑 2：Agent 抽取（適合直接餵原始財報）

新公司通常會讓關係抽取「解析不到節點」，所以要多跑一段「先建節點、再建關係」的流程：

```bash
cd ingestion

# 1. 第一次跑關係抽取（此時新公司還不存在，很多關係會解析不到）
python rag/extract.py              # 需 OPENAI_API_KEY
                                    # 找不到的實體會列在 rag/extracted/unresolved_entities.jsonl

# 2. 針對解析不到的實體，讓 LLM 建議新節點屬性
python rag/extract_entities.py     # 需 key
                                    # 可入庫 → rag/extracted/node_candidates.jsonl
                                    # 缺 ticker/exchange 等必要欄位 → rag/extracted/node_needs_manual_review.jsonl

# 3. 把節點候選寫入 Neo4j（status=candidate）
python rag/load_node_candidates.py

# 4. 到 /review 頁「候選節點」分頁先審核新公司節點（Accept）
#    （新節點存在後，關係抽取才能解析到它）

# 5. 重新對同一批 chunks 跑一次關係抽取（現在新公司節點已存在，可以解析成功）
python rag/extract.py

# 6. 把關係候選寫入 Neo4j
python rag/load_candidates.py

# 7. 到 /review 頁「候選關係」分頁審核
```

如果第 2 步某公司落在 `node_needs_manual_review.jsonl`（通常是缺 ticker / 交易所），
代表 Agent 沒把握組出正確 id，**改用上面「路徑 1：手動」在 `seed_companies.csv` 補上這家公司**，
之後重跑第 5-7 步即可正常解析。

## 3. 情境 B：已經登記過的公司（更新資料）

例如公司出了新年報，內容確認舊關係依然成立、或多了新的供應商 / 客戶、或某條舊關係已經不成立。

### 3.1 新財報「確認同一件事」→ 通常不用改舊資料

依 `docs/development/data-update-rules.md` 規則：**同一 from/to/type 若在新期間再次被確認，
建立一筆新的關係（新 period、新 source），舊的關係不動、不刪除**。做法與情境 A 的「路徑 2」相同，
只差公司節點已存在，可以跳過建節點的步驟：

```bash
cd ingestion
python rag/register_sources.py     # 登記這份新財報
python rag/parse_documents.py
python rag/extract.py              # 公司已存在，關係可直接解析成功
python rag/load_candidates.py      # 寫入新的 candidate 關係（新 period）
# 到 /review 審核
```

手動路徑：在 `seed_relationships.csv` 新增一列新的 `rel_id`（不要覆蓋舊列），
`source_ids` 指向新財報的 Source，`period` 填新的期間。

### 3.2 新財報「內容和舊資料矛盾」

不要直接覆蓋。依規則保留兩邊證據，交給人工審核判斷：

- 若只是來源更好（同期間內），且語意沒變（from/to/type 一樣）：可直接更新舊關係的
  `confidence` / `source_ids`（`updated_at` 一併更新）。
- 若語意變了（供應商換人、關係方向不同）：**不要改舊列**，用上面 3.1 的方式建新關係，
  舊關係之後視情況標記 `stale`（過期待確認）或 `deprecated`（確定不再成立）。

### 3.3 舊關係「已確定不再成立」（例如客戶轉單、供應商換掉）

目前系統的 `/review` API 只處理 `status=candidate` 的 Accept / Reject，
**還沒有現成的 API 可以把既有 `verified` 關係一鍵標成 `stale` / `deprecated`**。
現階段做法二選一：

1. 如果是 `manual_seed` 來源：直接在 `seed_relationships.csv` 找到該筆 `id`，
   把 `status` 改成 `deprecated`（或 `stale`），`note` 註明依據哪份新來源，
   重新執行 `python scripts/import_graph.py`（MERGE 會更新既有列，不會產生重複）。
2. 如果是已經 Accept 過的 candidate（`llm_extraction`）：可到 Neo4j Browser
   （http://localhost:7474）用 Cypher 手動更新該筆關係的 `status`，並記得寫
   `reviewed_at` / `reviewed_by` / `review_note`（欄位定義見 data-update-rules.md）。

**一律不要用 `DELETE` 刪除關係**（見 data-update-rules.md 規則 6），只能改 `status`。

## 4. 範例：手邊 5 家公司財報的完整流程

假設 3 家已登記（A、B、C）、2 家全新（D、E）：

```bash
cd ingestion

# 1. 檔案就位 + 登記 manifest（5 家財報各一列，company_id：A/B/C 填既有 id，D/E 留空）
#    編輯 rag/documents/manifest.csv

# 2. 登記 Source + 解析
python rag/register_sources.py
python rag/parse_documents.py

# 3. 第一輪抽取關係（A/B/C 可直接解析，D/E 的相關關係會落在 unresolved_entities.jsonl）
python rag/extract.py

# 4. 針對 D/E 建節點候選
python rag/extract_entities.py
python rag/load_node_candidates.py

# 5. 到 /review「候選節點」分頁，Accept D、E（confirm 沒有跟既有公司重複）

# 6. 重跑抽取（這次 D/E 也能解析成功），寫入關係候選
python rag/extract.py
python rag/load_candidates.py

# 7. 到 /review「候選關係」分頁逐筆審核（可先用 min_confidence 排序，高分先看）
```

審核時建議順序：**先審節點、再審關係**（關係一定依附在節點上，節點先確認身份正確，
關係的 from/to 才有意義）。

## 5. 檢查清單

- [ ] 每份財報都登記了 `Source`（`source_ids` 有東西可以指）
- [ ] 新公司 id 符合規則（`TWSE_xxxx` / `TPEX_xxxx` / `{國別}_{TICKER}`），沒有跟既有節點重複
- [ ] 關係方向正確（供應鏈方向不可顛倒，見 relationship-types.md）
- [ ] 沒有使用 `RELATED_TO` 等模糊關係
- [ ] `confidence` 依實際把握程度填，不是每條都給 0.9
- [ ] Agent 產出的資料都到 `/review` 過一遍才算數
- [ ] 沒有用 `DELETE` 刪除任何既有節點或關係
